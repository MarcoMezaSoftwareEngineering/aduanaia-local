DOCUMENTO TÉCNICO DEL PROYECTO
AduanaIA Local: Sistema inteligente de apoyo al análisis preliminar de mercancías restringidas, prohibidas e inmovilizadas
________________________________________
1. Nombre del proyecto
AduanaIA Local
Subtítulo técnico:
Sistema RAG con agentes inteligentes para apoyo a la determinación preliminar de mercancías restringidas, prohibidas e inmovilizadas en procesos aduaneros.
________________________________________
2. Resumen ejecutivo
El proyecto consiste en desarrollar un sistema local de inteligencia artificial orientado al campo aduanero, específicamente para apoyar el análisis preliminar de expedientes, mercancías, documentos y normativa relacionada con mercancías restringidas, prohibidas, inmovilizadas o sujetas a revisión documental.
El sistema no reemplaza al funcionario aduanero ni emite decisiones legales definitivas. Su función es actuar como asistente técnico que:
1.	Lee documentos aduaneros.
2.	Extrae información relevante del expediente.
3.	Busca normativa aplicable en una base documental local.
4.	Identifica posibles restricciones, prohibiciones o requisitos faltantes.
5.	Sugiere una recomendación preliminar sustentada.
6.	Genera un informe técnico para revisión humana.
El desarrollo debe realizarse priorizando herramientas gratuitas, open-source o sin licencias comerciales obligatorias, y debe ejecutarse localmente en una PC con GPU NVIDIA RTX 3080 de 10 GB de VRAM, evitando depender de APIs pagadas como OpenAI, Anthropic, Gemini, Azure o servicios cloud.
________________________________________
3. Problema que busca resolver
En el proceso aduanero, especialmente en regímenes como envíos de entrega rápida, importación para consumo, revisión documentaria o reconocimiento físico, el funcionario debe analizar información dispersa:
•	Descripción comercial de la mercancía.
•	Subpartida nacional o partida arancelaria.
•	Factura comercial.
•	Guía aérea o documento de transporte.
•	Declaración simplificada o DAM.
•	Actas de inmovilización.
•	Permisos sectoriales.
•	Normativa vigente.
•	Criterios internos.
•	Antecedentes o casos similares.
Este análisis consume tiempo y puede generar variabilidad en los criterios de revisión. Además, la normativa suele estar distribuida en múltiples documentos, resoluciones, procedimientos, tablas y comunicados.
El problema principal es que el funcionario o analista necesita una herramienta que le permita consultar rápidamente la normativa y obtener una orientación preliminar, trazable y explicable.
________________________________________
4. Objetivo general
Desarrollar un sistema local de inteligencia artificial basado en RAG y agentes, capaz de asistir en el análisis preliminar de mercancías aduaneras mediante consulta documental, recuperación normativa, evaluación de riesgo y generación de informes sustentados.
________________________________________
5. Objetivos específicos
1.	Construir una base documental local con normativa aduanera, procedimientos, catálogos de mercancías restringidas, resoluciones, criterios técnicos y casos anonimizados.
2.	Implementar un motor RAG que permita buscar información normativa relevante usando consultas en lenguaje natural.
3.	Desarrollar agentes especializados mediante LangGraph para dividir el análisis en etapas.
4.	Ejecutar el modelo de lenguaje localmente usando la GPU RTX 3080 de 10 GB.
5.	Generar respuestas fundamentadas, con citas internas del documento fuente, evitando respuestas inventadas.
6.	Permitir la carga de expedientes en PDF, texto o Excel.
7.	Generar un informe técnico preliminar exportable en PDF o Word.
8.	Registrar consultas, resultados, fuentes recuperadas y evaluación de confianza.

6. Alcance funcional del MVP
El MVP deberá permitir:
6.1. Carga documental
El usuario podrá subir:
•	PDF de expediente.
•	Factura comercial.
•	Guía aérea.
•	Acta de inmovilización.
•	Declaración simplificada.
•	Archivos Excel con relación de mercancías.
•	Texto manual con descripción comercial.
6.2. Extracción de datos
El sistema deberá extraer:
•	Nombre o descripción de la mercancía.
•	Cantidad.
•	Unidad de medida.
•	País de origen.
•	Valor declarado.
•	Subpartida, si existe.
•	Documentos presentados.
•	Documentos faltantes.
•	Observaciones relevantes.
•	Posible entidad competente.
6.3. Consulta normativa
El sistema deberá buscar en la base documental local:
•	Normas aduaneras.
•	Procedimientos de control.
•	Catálogo de mercancías restringidas.
•	Catálogo de mercancías prohibidas.
•	Procedimientos de inmovilización.
•	Requisitos sectoriales.
•	Antecedentes o casos similares.
•	Criterios técnicos previamente cargados.
6.4. Análisis preliminar
El sistema deberá clasificar el caso en alguno de estos niveles:
•	Riesgo bajo: no se detectan restricciones relevantes en la base local.
•	Riesgo medio: requiere revisión documental adicional.
•	Riesgo alto: posible mercancía restringida, permiso faltante o inconsistencia relevante.
•	Riesgo crítico: posible mercancía prohibida, causal de comiso, falsedad documental o imposibilidad de levante sin revisión humana.
6.5. Recomendación preliminar
El sistema podrá sugerir:
•	Continuar con revisión manual.
•	Solicitar documento faltante.
•	Validar autorización de entidad competente.
•	Mantener inmovilización.
•	Evaluar reembarque.
•	Derivar a especialista.
•	No emitir recomendación por falta de información.
6.6. Informe técnico
El sistema deberá generar un informe con:
1.	Resumen del expediente.
2.	Datos extraídos.
3.	Normativa relacionada.
4.	Fuentes consultadas.
5.	Análisis preliminar.
6.	Nivel de riesgo.
7.	Documentos faltantes.
8.	Recomendación sugerida.
9.	Advertencia de revisión humana obligatoria.

10. Modelo de IA recomendado para la RTX 3080 10 GB
Modelo principal recomendado
Qwen3 8B Instruct cuantizado
Formato sugerido:
•	GGUF Q4_K_M o Q5_K_M.
•	Ejecución mediante Ollama o llama.cpp.
•	Contexto recomendado inicial: 4096 a 8192 tokens.
•	Temperatura baja: 0.1 a 0.3.
•	Top-p: 0.8 a 0.9.
Alternativas
1.	Llama 3.1 8B Instruct cuantizado
2.	Mistral 7B Instruct cuantizado
3.	Gemma 3 4B/12B cuantizado, si el rendimiento es aceptable
4.	Phi-3.5 Mini, si se busca velocidad y bajo consumo
Criterio de selección
Para este proyecto no se recomienda usar modelos de 30B, 32B, 70B o superiores, porque la RTX 3080 de 10 GB no tiene VRAM suficiente para ejecutarlos cómodamente con buen rendimiento.
La prioridad es usar un modelo de 7B u 8B bien configurado, más una buena base documental RAG. En este caso, la calidad del sistema dependerá más de la recuperación correcta de documentos que del tamaño del modelo.

11. Modelo de embeddings recomendado
Opción recomendada
BGE-M3
Ventajas:
•	Funciona bien en entornos multilingües.
•	Adecuado para documentos en español.
•	Soporta textos relativamente largos.
•	Puede utilizarse para recuperación semántica.
•	Compatible con flujos RAG.
Alternativa ligera
nomic-embed-text
Ventajas:
•	Fácil de usar con Ollama.
•	Menor consumo de recursos.
•	Adecuado para un MVP rápido.
Recomendación práctica
Para el MVP:
•	Usar nomic-embed-text si se prioriza simplicidad.
•	Usar BGE-M3 si se prioriza calidad de recuperación documental.

•	Texto manual

  "tipo_documento": "factura / guia / acta / expediente",
  "calidad_extraccion": "alta / media / baja",
  "requiere_ocr": true
}
________________________________________
15.2. Agente extractor estructurado
Responsable de convertir texto no estructurado en datos ordenados.
Salida esperada:
{
  "mercancia": "suplemento alimenticio",
  "descripcion_comercial": "frascos de cápsulas para consumo humano",
  "cantidad": "120",
  "unidad": "frasco",
  "pais_origen": "Estados Unidos",
  "subpartida": "no identificada",
  "documentos_presentados": [
    "factura comercial",
    "guia aerea"
  ],
  "documentos_faltantes_posibles": [
    "autorizacion sanitaria",
    "registro sanitario"
  ]
}
________________________________________
15.3. Agente de recuperación normativa
Responsable de consultar Milvus y recuperar fragmentos relevantes.
Debe buscar por:
•	Descripción de mercancía.
•	Sinónimos.
•	Subpartida.
•	Entidad competente.
•	Términos normativos.
•	Documentos requeridos.
Salida esperada:
{
  "fuentes_recuperadas": [
    {
      "documento": "Catalogo de mercancias restringidas",
      "fragmento": "...",
      "score": 0.83,
      "pagina": 12
    }
  ]
}
________________________________________
15.4. Agente evaluador de mercancía
Responsable de analizar si la mercancía podría ser:
•	Libre.
•	Restringida.
•	Prohibida.
•	Sujeta a validación.
•	Sujeta a revisión por entidad competente.
Salida esperada:
{
  "clasificacion_preliminar": "posible mercancia restringida",
  "entidad_competente_probable": "DIGEMID / SENASA / SUCAMEC / MTC / PRODUCE / MINSA / otra",
  "fundamento": "La descripción se relaciona con productos de consumo humano...",
  "nivel_confianza": "medio"
}
________________________________________
15.5. Agente evaluador documental
Responsable de comparar documentos presentados vs. documentos requeridos.
Salida esperada:
{
  "documentos_presentados": [
    "factura",
    "guia aerea"
  ],
  "documentos_requeridos_probables": [
    "autorizacion sanitaria"
  ],
  "brecha_documental": true,
  "observacion": "No se encontró autorización sectorial en los documentos cargados."
}
________________________________________
15.6. Agente clasificador de riesgo
Responsable de asignar un nivel de riesgo.
Reglas iniciales:
Riesgo bajo:
- No se detecta restricción.
- Documentos aparentemente completos.
- Fuentes recuperadas con baja alerta.

Riesgo medio:
- Existe incertidumbre.
- Falta información.
- La mercancía podría requerir validación adicional.

Riesgo alto:
- Se encuentra coincidencia con mercancía restringida.
- Falta autorización probable.
- Hay inconsistencias entre descripción y documentos.

Riesgo crítico:
- Se encuentra coincidencia con mercancía prohibida.
- Hay posible causal de comiso.
- Hay indicios de falsedad, ocultamiento o contradicción grave.
________________________________________
15.7. Agente redactor de informe
Responsable de redactar un informe claro, técnico y no vinculante.
Debe incluir:
•	Resumen.
•	Datos extraídos.
•	Normativa consultada.
•	Análisis.
•	Riesgo.
•	Recomendación.
•	Advertencia de revisión humana.

18. Pipeline de ingesta documental
Paso 1: Carga
El usuario o administrador carga documentos base:
•	Normativa.
•	Procedimientos.
•	Catálogos.
•	Resoluciones.
•	Casos anonimizados.
Paso 2: Extracción
Se usa:
•	PyMuPDF para PDF digital.
•	pdfplumber para tablas.
•	Tesseract o PaddleOCR para PDF escaneado.
Paso 3: Limpieza
Eliminar:
•	Encabezados repetidos.
•	Pies de página.
•	Numeración irrelevante.
•	Caracteres rotos.
•	Saltos excesivos.
Paso 4: Chunking
Recomendación inicial:
chunk_size: 800 a 1200 tokens
chunk_overlap: 100 a 200 tokens
Para normativa, debe evitarse cortar artículos legales a la mitad. Si es posible, el chunk debe respetar:
•	Artículo.
•	Numeral.
•	Inciso.
•	Tabla.
•	Anexo.
Paso 5: Embeddings
Generar embeddings con BGE-M3 o nomic-embed-text.
Paso 6: Indexación
Guardar:
•	Texto completo en PostgreSQL.
•	Embeddings en Milvus.
•	Metadatos en ambas bases.

Analiza un expediente o descripción de mercancía.

{
  "description": "suplemento alimenticio en capsulas",
  "tariff_code": "",
  "country_origin": "Estados Unidos",

  "report_id": "uuid"

GET /documents
Lista documentos cargados.
DELETE /documents/{id}
Elimina documentos de PostgreSQL y Milvus.

32. Recomendaciones de rendimiento para RTX 3080 10 GB
1.	Usar modelo 7B u 8B cuantizado.
2.	Preferir cuantización Q4_K_M o Q5_K_M.
3.	Limitar contexto inicial a 4096 o 8192 tokens.
4.	Evitar cargar modelos grandes.
5.	Usar batch pequeño.
6.	No ejecutar OCR masivo mientras el LLM está generando.
7.	Mantener Milvus Lite local.
8.	Procesar documentos en cola.
9.	Usar streaming de respuesta.
10.	Monitorear VRAM con nvidia-smi.

33. Riesgos técnicos
Riesgo 1: Alucinaciones del modelo
Mitigación:
•	Restringir respuesta a fuentes recuperadas.
•	Exigir citas internas.
•	Usar temperatura baja.
•	Mostrar advertencia cuando no hay evidencia.
Riesgo 2: Mala extracción de PDFs escaneados
Mitigación:
•	Usar OCR.
•	Mostrar calidad de extracción.
•	Permitir corrección manual.
Riesgo 3: Recuperación normativa incorrecta
Mitigación:
•	Mejorar chunking.
•	Usar metadatos.
•	Probar BGE-M3.
•	Agregar reranker si es necesario.
Riesgo 4: Falta de VRAM
Mitigación:
•	Usar modelo más pequeño.
•	Reducir contexto.
•	Usar cuantización Q4.
•	Evitar modelos mayores a 8B.
Riesgo 5: Base normativa desactualizada
Mitigación:
•	Registrar fecha de carga.
•	Crear módulo de actualización.
•	Mostrar versión documental usada.

34. Recomendación final de implementación
Para iniciar, se recomienda implementar el proyecto en este orden:
1.	RAG simple con documentos normativos.
2.	Consulta en lenguaje natural.
3.	Carga de expediente PDF.
4.	Extracción estructurada.
5.	Evaluación de riesgo.
6.	LangGraph con agentes.
7.	Informe exportable.
8.	Validación con casos reales anonimizados.
No se recomienda iniciar con una arquitectura demasiado grande. El valor principal del MVP está en demostrar que el sistema puede recuperar normativa correcta, analizar un caso y generar una recomendación preliminar sustentada.

35. Producto final esperado
El producto final del MVP será una aplicación local llamada AduanaIA Local, ejecutada en una PC con RTX 3080 de 10 GB, que permita cargar documentos aduaneros, consultar normativa, analizar expedientes, detectar posibles restricciones o documentos faltantes y generar informes preliminares sustentados para revisión humana.
La IA debe funcionar como asistente técnico, no como autoridad decisoria.
